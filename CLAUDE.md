# CLAUDE.md

## 変更履歴の記録ルール

記録は2つのファイルに分ける。どちらも新しい更新が上（先頭）に来るようにする。

- **CHANGELOG.md** — 要点だけのサマリ。一覧して「何をしたか」がすぐ分かるようにする
- **NOTES.md** — 作業メモ。実装方式・検証手順・再現手順・実行したプロンプト全部などの詳細はこちらに書く

### CHANGELOG.md の書き方

- commit / push したら記録する
- **1エントリは最大8行**。以下のテンプレートに従い、簡潔に書く。

```markdown
## YYYY-MM-DD タイトル（何をしたか一言で）
- 変更点を箇条書きで1〜3行（対象ファイル名を含める）
- プロンプト: 「ユーザーが指示した主要プロンプト1〜2個」
```

**書くこと**

- 何をしたか（結果）と、対象ファイル
- 再現の起点になる主要プロンプト（最大2個）
- URLやコマンドなど、後から git log では分からない情報（デプロイURL等）

**書かないこと（NOTES.md に書く）**

- 実装方式の詳細な解説
- 検証手順・再現手順の詳細
- 機能の網羅的なリストアップ（要点1〜3個に絞る）

### NOTES.md の書き方

- CHANGELOG.md と同じ日付・タイトルでエントリを作り、対応が分かるようにする
- 行数制限なし。後で見て作業を再現できるように、実装方式・検証内容・実行したプロンプト全部を残す
- commit / push したら記録する

## commit / push 前の安全確認

commit / push の前に、必ず以下を確認する。

- `git status` と `git diff --staged` を確認する
- 意図したファイルだけが対象になっていることを確認する
- `.gitignore` がプロジェクト構成に合っていることを確認する
- `.env`、APIキー、パスワード、トークン、秘密鍵、認証情報、個人情報が含まれていないことを確認する
- `node_modules` や不要な生成ファイルが含まれていないことを確認する
- 可能な場合は、動作確認・ビルド・テストを行う

秘密情報を発見した場合は、commit / push を中止する。

- 秘密情報の値そのものは表示・記録しない
- 対象ファイル名と問題の種類だけを報告する
- 必要に応じて `.gitignore` へ追加する
- すでにGit履歴へ入っている可能性がある場合は、その可能性を伝える
- CHANGELOG.md、NOTES.md、README.mdにも秘密情報の実値を書かない
